# 三公鲜配极光基础推送设计

日期：2026-07-10

## 目标

为 Android App 增加订单通知能力：

- 管理员在子单位提交订单后收到新订单提醒。
- 子单位在订单进入已接单、备货中、已发货、已完成或已取消时收到状态提醒。
- Android 13 及以上系统明确申请通知权限，用户拒绝后可以从“我的”页面恢复授权。
- 推送失败有重试和日志，业务事务失败时不得发送通知。
- 所有通知点击后仍由服务端鉴权，不能依赖通知载荷决定角色、单位或订单权限。

本阶段只接入极光基础通道，不配置华为、小米、荣耀、OPPO、vivo 等厂商离线通道。

## 已确认限制

极光基础通道依赖 SDK 长连接。App 正在运行或后台连接仍然存活时可以实时送达；进程被手机系统彻底清理后，不保证实时送达。因此本阶段增加前台同步和低频后台同步作为补偿，但不能把它描述成全品牌可靠离线推送。

后续补充厂商通道时，复用本设计的设备绑定、推送事件和服务端发送接口，不修改订单业务规则。

## 安全边界

- Android APK 只包含极光 `AppKey`。`AppKey` 与包名 `com.smartprocurement.internal` 保持一致。
- 极光 `Master Secret` 只保存在生产服务器 `.env`，不得写入 APK、源码、GitHub、数据库日志或客户端错误日志。
- 已经通过截图或其他非安全渠道展示过的 `Master Secret` 必须先在极光控制台重置，本项目只使用重置后的新密钥。
- 服务端设备绑定从当前登录 Token 获取 `user_id`、`role` 和 `unit_id`，不信任客户端上传的角色或单位。
- 推送正文不包含单位名称、金额、食材明细、联系人或配送点，只使用中性文案和订单编号。
- 通知载荷只携带随机 `event_id`、`event_type` 和 `order_id`。打开通知后 App 使用当前登录 Token 请求订单详情，服务端继续执行管理员或单位归属校验。
- 退出登录、账号停用、单位停用、密码修改导致会话失效时，服务端停止向相关设备发送业务推送。

## Android 端设计

### 权限和频道

- Manifest 增加 `android.permission.POST_NOTIFICATIONS`。
- Android 13 及以上在用户成功登录并确认进入业务系统后申请通知权限，不在启动页无说明弹出。
- 创建两个通知频道：
  - `new_orders`：新订单提醒，高重要性，供管理员使用。
  - `order_updates`：订单进度提醒，默认重要性，供子单位使用。
- “我的”页面显示“订单通知”状态：已开启、未开启。未开启时提供明确按钮跳转系统通知设置。

### 极光注册

- 用户同意必要隐私说明并登录成功后初始化极光 SDK。
- SDK 返回 `Registration ID` 后，调用服务端设备注册接口。
- Registration ID 变化时重新注册，服务端以设备安装标识和 Registration ID 幂等更新。
- 退出登录时调用设备解绑接口；解绑失败不阻止本地退出，服务端会话失效仍会阻止后续业务访问。

### 通知点击

- 通知点击打开 `MainActivity`，携带 `order_id` 和 `event_id`。
- App 完成登录态和首次改密检查后，再进入订单详情。
- 未登录时进入登录页；登录成功后仅在当前账号有权查看该订单时进入详情。
- 同一 `event_id` 只处理一次，避免重复打开和重复提示。

### 补偿同步

- App 进入前台时继续调用现有订单刷新逻辑。
- 使用 WorkManager 进行不高于系统允许频率的低频后台同步，建议 15 分钟一次，并要求有网络连接。
- 同步发现此前未提示的新订单或状态变化时生成本地通知。
- 本地保存最后已通知的订单版本或事件编号，避免极光通知和后台同步重复提醒。

## 服务端设计

### 数据表

新增迁移，建立：

1. `push_devices`
   - `id`
   - `user_id`
   - `registration_id`
   - `installation_id`
   - `platform`
   - `app_version`
   - `active`
   - `last_seen_at`
   - `created_at`
   - `updated_at`

2. `push_outbox`
   - `id`，同时作为 `event_id`
   - `event_type`
   - `order_id`
   - `recipient_scope`
   - `recipient_unit_id`
   - `payload_json`
   - `status`
   - `attempt_count`
   - `next_attempt_at`
   - `sent_at`
   - `last_error`
   - `created_at`

3. `push_deliveries`
   - `id`
   - `event_id`
   - `device_id`
   - `provider_message_id`
   - `status`
   - `attempt_count`
   - `last_error`
   - `created_at`
   - `updated_at`

`event_id + device_id` 建立唯一约束，防止重复发送。

### API

- `POST /api/v1/push/devices/register`
  - 需要有效 App Bearer Token。
  - 接收 Registration ID、安装标识、App 版本。
  - 用户、角色和单位从服务端会话读取。

- `DELETE /api/v1/push/devices/current`
  - 需要有效 App Bearer Token。
  - 停用当前安装对应的设备绑定。

- `POST /api/v1/push/events/{event_id}/opened`
  - 记录通知点击结果，用于排查未响应和重复点击。

上述接口统一返回中文安全错误，不返回极光内部错误、密钥或 Python 异常。

### 事件产生

- 子单位订单创建事务成功时，在同一数据库事务写入 `ORDER_CREATED` outbox 事件，收件范围为所有启用管理员。
- 订单状态事务成功时写入 `ORDER_STATUS_CHANGED` 事件，收件范围为订单所属单位的启用账号。
- 状态文案映射：
  - `accepted`：订单已接单。
  - `preparing`：订单正在备货。
  - `shipped`：订单已发货，请及时确认收货。
  - `completed`：订单已完成。
  - `cancelled`：订单已取消。
- 事务回滚时 outbox 事件同时回滚，不能出现订单未成功但通知已发送。

### 推送任务

- 独立推送任务读取待发送 outbox，不阻塞订单接口响应。
- 服务端使用 `JPUSH_APP_KEY` 和 `JPUSH_MASTER_SECRET` 调用极光 REST API。
- 收件设备发送前重新检查账号启用状态、单位启用状态和角色。
- 失败采用有限指数退避，达到上限后标记失败并写入现有系统日志页面。
- 极光返回无效 Registration ID 时停用对应设备记录。
- 日志只记录事件编号、设备记录 ID、结果和脱敏错误，不记录完整凭据。

## 通知文案

管理员新订单：

- 标题：`三公鲜配 · 新订单`
- 正文：`收到新的采购订单，点击查看。`

子单位状态更新：

- 标题：`三公鲜配 · 订单进度`
- 正文：按服务端状态映射显示，不展示其他业务敏感字段。

系统通知权限关闭：

- 页面状态：`订单通知未开启`
- 操作按钮：`开启订单通知`

## 错误与恢复

- 用户拒绝通知权限：业务功能继续可用，“我的”页面持续显示未开启状态。
- 极光 SDK 注册失败：静默重试并写入本地诊断日志，不阻止登录和下单。
- 设备注册接口失败：保留 Registration ID，下一次进入前台或后台同步时重试。
- 极光发送失败：由 outbox 重试，不回滚已经成功的订单业务。
- 登录失效：停止后台业务同步，通知点击进入登录页。
- 重复推送：服务端唯一约束和客户端事件去重共同处理。

## 测试范围

### Android 单元与 Robolectric 测试

- Android 13 以上需要通知权限，旧版本不需要运行时申请。
- 管理员只对 `ORDER_CREATED` 生成新订单通知。
- 子单位只对自己订单的状态变化生成进度通知。
- 同一事件不重复通知。
- 点击通知在未登录、首次改密和正常登录三种状态下正确分流。
- 通知关闭状态和系统设置入口可用。
- 后台同步失败不显示假通知。

### 服务端测试

- 注册设备只绑定当前登录用户。
- 修改请求中的角色或单位字段不能改变绑定结果。
- 新订单事务写入管理员事件。
- 状态变化只写入订单所属单位事件。
- 订单事务回滚不产生事件。
- 停用账号和停用单位不接收推送。
- 同一事件和设备不会重复发送。
- 极光失败会重试并脱敏记录错误。
- Master Secret 不出现在 API、日志、测试快照和导出文件中。

### 真机验收

- Android 13 及以上首次登录出现通知权限申请。
- 管理员前台、后台收到新订单通知。
- 子单位前台、后台收到订单状态通知。
- 点击通知进入正确订单。
- 拒绝权限后可从“我的”恢复。
- App 被系统彻底清理后的送达情况单独记录为极光基础通道限制，不作为厂商离线通道已完成的证据。

## 部署前置条件

- 在极光控制台重置已经暴露的 Master Secret。
- 极光应用包名保持 `com.smartprocurement.internal`。
- 将新的 `JPUSH_MASTER_SECRET` 仅写入生产服务器 `.env`。
- Android 构建通过 Gradle 配置注入 `JPUSH_APP_KEY`，不在页面或日志显示。
- 不修改 Android applicationId、现有数据库名称、API 前缀和订单状态机。
- 发布 APK 前保持签名兼容，并对当前正式签名做单独核验。

## 不在本阶段范围

- 华为、小米、荣耀、OPPO、vivo、魅族和 FCM 厂商离线通道。
- 短信、微信、邮件通知。
- 消息中心、营销推送和广播通知。
- 在锁屏通知中显示订单金额、单位名称或食材明细。
